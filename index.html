IT'S ALL ABOUT REPOS

Primeros pasos

Inicializar un repositorio 

git init

Clonar un repositorio 

git clone url_repositorio
- qué es un repositorio - .git - remote.origin.url

Git reconoce el proyecto en el que estás trabajando en función del lugar donde te encuentres al ejecutar los comandos

Mantener los archivos de nuestro repositorio

git add nombre_archivo

Eliminar un archivo del repositorio
- para que git sepa que debe de empezar a tener en cuenta un archivo

Mantener los archivos de nuestro repositorio

git status
git commit nombre_archivo
git commit -m "Comentario de los archivos"
git commit -am "Comentario de los archivos"

- COMMIT envía del INDEX al HEAD

- -a: conviene su uso ya que examina todos los ficheros que están siendo seguidos, no sólo los del directorio actual y los que están por debajo

Sincronizando repositorios 

git remote
git remote -v 
origin git@github.com:cecimerelo/GIT.git (fetch) 
origin git@github.com:cecimerelo/GIT.git (push)

- git remote los repositorios que tenemos en remoto

- recibir(fecth) y enviar(push) los cambios

Sincronizando repositorios

Añadir otro repositorio remoto: 

git remote add ALIAS DIRECCION

Eliminar repositorio remoto:

git remote rm NOMBRE

Cambiar el nombre:

git remote rename NOMBRE_ANTERIOR NOMBRE_NUEVO

- alias es un nombre corto para usar en las instrucciones

- git no comprueba si realmente existen los repositoriosque agregas o si tienes permisos de lectura o escritura en ellos. Que estén ahí no significa que puedas usarlos realmente.

Recibiendo cambios 

git pull REPOSITORIO_REMOTO RAMA
git pull origin master
git pull
git pull = git fetch + git merge

- importar cambios desde un repositorio remoto

- si el repositorio del que tratas de importar no existe o no tiene permisos de lectura, te dará un mensaje de error.

Enviando cambios 

git push REPOSITORIO_REMOTO RAMA

- se recomienda hacer un push por cada commit, por si alguien está modificando el mismo código.

- lo primero es un hash unico

- nos mostrará todas las entradas

- --oneline mostrará una versión resumida, al contrario que -p

- git log --graph --oneline commits en versión compacta y dibujando las ramas

Borrado de archivos

git rm NOMBRE_FICHERO
git rm --cached NOMBRE_ARCHIVO
git reset HEAD NOMBRE_ARCHIVO

- borra también el archivo del index

- --cached: necesitas borrar el archivo del index pero sin borrarlo del directorio de trabajo

Retrocediendo al pasado

git commit --amend
git checkout --NOMBRE_ARCHIVO
git reset --hard HASH_COMMIT
git show HASH_COMMIT:ruta/archivo

- --amend : si no has modificado nada en tus archivos esto simplemente te permite reescribir el commit, pero si habías olvidado añadir algo al index lo puedes añadir antes de usar esta orden

- --checkout : deshacer cambios en un archivo, para recuperar la versión del HEAD

- (3) para recuperar el estado de tu directorio de trabajo tal y como estaba en algún momento del pasado. Deshara los commits posteriores al indicado y traerá a tu directorio de trabajo los archivos tal y como estaban entonces.

- usar push antes del reset

- usar ramas

- (4) cambios que hiciste en un commit

Buenas prácticas

No hay reglas de cómo debe de hacerse pero sí de cómo NO debe de hacerse

Ficheros en el directorio principal: INSTALL, .gitignore, LICENSE, makefile ...

- todo en un único directorio. Cada persona sólo vea los arhivos con los que tiene que trabajar.

- incluir en la cabecera en qué paquete se encuentra y su licencia

- fuera de los mencionados se intentará no poner nada más en el directorio principal

Ramas ligeras

git tag v0.0.2
git tag -a v0.0.2 -m "Comment"
git show v0.0.2

- permite guardar el estado de un respositorio en un momento determinado

- tag etiqueta el último commit, asigna una etiqueta al estado en el que estaba el repositorio tras el último commit

- -a: añade una etiqueta a una rama

Ramas

Ejemplo de flujo: 

git branch NOMBRE_RAMA
git checkout NOMBRE_RAMA

Aquí se hacen cosas

git checkout master
git merge NOMBRE_RAMA
git branch -d NOMBRE_RAMA

- una rama es como un proyecto diferente que surge de un proyecto principal. Son caminos divergentes a partir de un tronco común, que eventualmente pueden combinarse en uno sólo.

- cada usuario trabaja con su propia rama
